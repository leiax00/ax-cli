# Smart Commit

将 `.claude/rules/git-commit.md` 转换为可执行的提交技能流程。

## 何时启用

- 用户明确表达要提交代码（例如：`提交代码`、`commit`）

## 提交时机（强制）

- 只有在用户明确请求后，才允许创建提交
- 禁止主动创建提交
- 禁止询问“是否需要提交”
- 禁止暗示或建议提交

## 提交前检查（强制）

创建提交前，必须按顺序执行：

1. `git status`（查看工作区状态）
2. `git diff`（审查改动内容）
3. `git log -5 --oneline`（对齐项目提交风格）
4. 使用 `code-reviewer` 智能体完成代码审查

## 提交信息格式（强制）

统一格式：

`<type>(<scope>): <subject>`

### type（类型）

| 值 | 说明 |
|---|---|
| `feat` | 新功能 |
| `fix` | 修复 Bug |
| `improve` | 功能改进（非新功能） |
| `opt` | 代码优化（逻辑简化、可读性提升） |
| `refactor` | 重构 |
| `perf` | 性能优化 |
| `style` | 代码格式（不影响功能） |
| `test` | 测试相关 |
| `docs` | 文档 |
| `chore` | 构建/工具链 |

### scope（范围）

- 必填，表示改动所属模块
- 推荐使用简短模块名（如 `device`、`upgrade`、`cache`）

### subject（主题）

- 中文描述
- 不超过 50 字符
- 祈使句（动词开头：增加、修复、更新、优化、重构）
- 不以句号结尾

示例：

- `feat(device): 增加设备批量导入`
- `fix(cache): 修复策略缓存失效条件`

反例：

- `feat(device): 增加了设备批量导入`
- `feat(device): 增加设备批量导入功能。`

## 历史修改操作（谨慎）

执行以下操作前，必须先做风险沟通：

- `git commit --amend`
- `git rebase -i`
- `git reset --soft`

前置要求：

1. 告知用户该操作影响
2. 确认已推送提交是否受影响
3. 警告历史丢失风险

## 执行流程（Checklist）

1. 校验用户是否明确请求提交
2. 完成提交前检查四步
3. 生成符合规范的提交信息
4. 创建提交
5. 返回结果（提交哈希 + 提交信息）
